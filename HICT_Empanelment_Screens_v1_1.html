🔧 DEVELOPER NOTE — NOT PART OF UI · MODULE OVERVIEW (read this first)

The Empanelment module tracks a payor agreement from initiation to activation, and then through its live life-cycle (Active → Renewal Due → Expired). It sits alongside the existing collection workflow and reuses HICT's existing role, unit, and payor-category infrastructure. Use the "view as role" selector in the sidebar — action panels and buttons in every screen change with the selected role. That selector is a mockup device only; in production the role comes from the JWT.

New role: MARKETING_ASSOCIATE
  • New user role added to the existing role enum. Profile fields captured at user creation (Admin → User Management): name_as_per_hmis, employee_id, phone, official_email, department_as_per_hmis, designation_as_per_hmis, unit.
  • Marketing Associate sees only the Empanelment module — no access to bills, collections, or any existing screens.
Stage model & status enum
EMP_STATUS: STAGE_1_MGMT_REVIEW "Management Review" owner: MANAGEMENT (initiating unit) STAGE_2_FIN_REVIEW "Finance Review" owner: FINANCE STAGE_3_GROUP_REVIEW "Group Management Review" owner: GROUP_MANAGEMENT ⚠ see below STAGE_4_LEGAL_DRAFTING "Agreement Drafting — Legal" owner: MARKETING_ASSOCIATE (updates status) STAGE_5_WITH_PAYOR "With Payor — Signature Pending" owner: MARKETING_ASSOCIATE STAGE_6_ACTIVATION "Activation Pending — Billing" owner: BILLING (empanelment's unit) ACTIVE terminal-live RETURNED_FOR_MODIFICATION owner: MARKETING_ASSOCIATE (resubmit → restarts at Stage 1) REJECTED terminal (only from Stages 1-3) EXPIRED derived (expiry_date < today), set by nightly job RENEWAL_DUE ⚠ NOT a stored status — a derived view filter: status = ACTIVE AND expiry_date <= today + 45 days
✅ CONFIRMED — Group Management write exception
  • GROUP_MANAGEMENT stays view-only everywhere else in HICT. For this module only, a scoped permission EMPANELMENT_APPROVER is granted to designated Group Management users — this is the single action Group Management holds inside HICT. @PreAuthorize on the Stage-3 decision endpoint checks this permission, not the role. All other write endpoints continue to reject GROUP_MANAGEMENT unconditionally.
✅ CONFIRMED — Empanelment is unit-wise
  • Every empanelment belongs to exactly one unit (stamped from the initiator's profile), and the entire journey runs unit-wise: Stage 1 review by that unit's Management, Stage 6 activation undertaking by that unit's Billing team only.
  • The Marketing Associate role is unit-wise — each Marketing Associate belongs to one unit and sees only that unit's empanelments. Finance and Group Management see all units.
  • Same payor to be empanelled at another unit → a separate empanelment record with its own journey.
✅ CONFIRMED — Renewal
  • Initiate Renewal is in scope for v1: it opens the Initiate form pre-filled from the expiring/expired empanelment, links the records via renewal_of_id, and the renewal runs the same full Stage 1–6 journey as a fresh empanelment — no shortcuts.

🔧 DEVELOPER NOTE — NOT PART OF UI · STATE MACHINE & GUARDS

Transitions (server-enforced, @PreAuthorize by role + stage-owner check): STAGE_1 --approve(MANAGEMENT, unit match)--> STAGE_2 STAGE_2 --approve(FINANCE)---------------> STAGE_3 STAGE_3 --approve(EMPANELMENT_APPROVER)--> STAGE_4 STAGE_4 --legal_ok(MARKETING, owner)-----> STAGE_5 STAGE_5 --signed(MARKETING, owner, requires: agreement_doc + start_date + expiry_date)--> STAGE_6 STAGE_6 --activation confirmed(BILLING, unit match)--> ACTIVE STAGE_1..5 --return(stage owner, remarks mandatory)--> RETURNED_FOR_MODIFICATION STAGE_1..3 --reject(stage owner, remarks mandatory)--> REJECTED (terminal) (Reject is NOT available at Stages 4-6 — confirmed) RETURNED --resubmit(MARKETING, owner)---> STAGE_1 (cycle_no += 1)
  • Reject vs Return (✅ confirmed): Reject is available only at the review stages 1–3 and is terminal; Return for Modification is available at Stages 1–5. Both require mandatory remarks. At Stages 4–5 the only exits are progress or Return.
  • Return remarks + "terms to be changed" flag are stored on the activity row and pinned on the resubmission form.
  • Editing the empanelment is allowed only in RETURNED_FOR_MODIFICATION status, only by the initiating Marketing Associate. Snapshot terms per cycle in the activity log so reviewers can diff.
  • Stage 6: PATCH /api/empanelments/{id}/activate — allowed only for BILLING users of the empanelment's unit; writes empanelment_activation, flips status to ACTIVE, stamps activated_at.

ℹ️ How Empanelment TAT works

Enter the maximum number of working days (Monday–Friday, excluding Saturday and Sunday) an empanelment should remain in each stage per payor category. If an empanelment exceeds this threshold, a TAT Breached badge appears across the module — pipeline, detail, dashboard and reports. From 80% of the TAT it shows amber.
Leave a cell blank to disable TAT tracking for that combination.
